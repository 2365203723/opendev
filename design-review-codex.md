## VERDICT: BLOCK
The architecture direction is broadly right: reuse Cairn’s Fact/Intent/Hint model, keep SQLite+WAL, and strip Docker/multi-worker complexity for a single-process CLI. I would still block implementation start until the bootstrap path, hint transport, scope enforcement, and acceptance mismatch are corrected.

## CRITICAL (must fix before dev starts)
- The design removes Cairn’s explicit Bootstrap task without replacing it with a deterministic first action. Cairn’s own flow is `Bootstrap -> Reason -> Explore`, not just `Reason -> Explore` ([Cairn README](https://github.com/oritera/Cairn) lines 293-299; ADR lines 52-59). In PentAGI, the OODA loop starts from an empty snapshot and `Act` breaks if no open intent exists, without setting a terminal state ([design.md](E:\projects\PentAGI\doc\design.md:251) lines 251-317). That means AC3 and AC6 are not deterministic, and the scan can exit with zero Facts or a task left as `running`. Fix: create a bootstrap intent at task creation (`nmap -sV` at minimum, optionally `httpx` for HTTP services), or hard-code the first Act step before LLM planning.
- The hint queue is not safe or idempotent. `pentagi hint` writes both DB and `hints_pending.json`, then `Orient` reads the file, writes to DB again, and clears the file ([design.md](E:\projects\PentAGI\doc\design.md:319) lines 319-324). That creates duplicate hints by design, and it can also lose hints written between “read” and “clear.” `append + flock` is also not a valid protocol for a JSON document. Fix: choose one authority. The simplest design is DB-only queueing under SQLite WAL; if wake-up is needed, use a touch file/event, not a second data store. If a file queue is retained, make it JSONL or spool files with atomic rename plus dedupe keys.
- The security boundary exists only in prompts, not in the tool layer. The system prompt says “Authorized targets: only the target specified by the user” ([design.md](E:\projects\PentAGI\doc\design.md:406) lines 406-417), but the actual tool builders shown do not enforce hostname/scheme/path scope for `httpx`, `gobuster`, or `sqlmap` ([design.md](E:\projects\PentAGI\doc\design.md:366) lines 366-390). A prompt-injected page title, directory name, or sqlmap log line can push the LLM toward arbitrary URLs or hosts. Fix: derive a canonical allowlist from the task target, enforce host equality/same-origin at the tool layer, reject non-`http/https` URLs, reject non-canonical IP/host pivots, and never trust prompt instructions for scope control.
- `task_id` is used to build filesystem paths for `hint`, `report`, and `stop`, but the design never requires ULID validation or path normalization ([design.md](E:\projects\PentAGI\doc\design.md:333) lines 333-337). If the implementation does a naive path join, `../../...` traversal is possible. Fix: require `task_id` to match the ULID format, load the task from DB first, build paths from the persisted task directory, and reject any resolved path outside the root task directory.
- Acceptance is misaligned with product scope. Charter AC4 only verifies report headings plus one real Fact row ([charter.md](E:\projects\PentAGI\doc\charter.md:49) lines 49-60); it does not prove vulnerability discovery beyond port scanning. The current system can pass all 10 ACs without ever creating a `vuln` Fact. Because the acceptance target is only `scanme.nmap.org`, nothing in the design makes non-port vulnerability findings deterministic. Fix either the charter or the target: add an AC that requires at least one non-port enrichment/finding on a safe purpose-built target, or narrow the scope text so it no longer promises vulnerability discovery in MVP acceptance.

## HIGH (should fix)
- The end-of-iteration snapshot dump is stale. `snapshot` is captured before `Orient/Decide/Act`, but that same object is dumped after new facts/intents are written ([design.md](E:\projects\PentAGI\doc\design.md:265) and [design.md](E:\projects\PentAGI\doc\design.md:309)). Mid-run `blackboard.json` will lag one iteration behind the DB, and crash recovery/debugging will miss the latest work. Rebuild the snapshot after `Act` before dumping.
- The schema has integrity gaps and internal drift. The prose says `target_url` and `pen_facts` were added, but the DDL has neither ([design.md](E:\projects\PentAGI\doc\design.md:105) vs. [design.md](E:\projects\PentAGI\doc\design.md:113)). `intents.to_fact_id` has no composite FK back to `(facts.id, task_id)`. `tasks` lacks `stop_reason`/`failure_reason` even though the pseudocode passes reasons into `set_status(...)` ([design.md](E:\projects\PentAGI\doc\design.md:257), [design.md](E:\projects\PentAGI\doc\design.md:315)). Add FKs and CHECKs for non-negative iter/token counters and `confidence BETWEEN 0 AND 1`; add reason columns if stop/fail reasons matter operationally.
- The error-code matrix is underspecified. `scan` maps only `2/3/4/5`, and `hint/report/stop` only cover `2/6/7` ([design.md](E:\projects\PentAGI\doc\design.md:332) lines 332-339). Missing cases include Anthropic/API failure, DB lock/corruption, invalid task ID format, unsafe tool args, report output write failure, snapshot write failure, malformed parser output, and interrupted/STOP exits. Define a complete, closed table in `errors.py` before implementation.
- Snapshot truncation is too naive for a fact-graph system. “Latest 50 facts + open intents + unconsumed hints” ([design.md](E:\projects\PentAGI\doc\design.md:426) lines 426-429) can easily drop the early service map once gobuster/httpx floods the board, which means the LLM loses the causal facts that later intents depend on. Keep a durable target summary and all facts referenced by open intents/concluded findings; do not rely on simple recency alone.
- SQLMap hardening should be a positive allowlist, not only a forbidden-flag set. The current spec blocks a subset of dangerous flags ([design.md](E:\projects\PentAGI\doc\design.md:375) lines 375-390) but does not spell out host validation, allowed schemes, redirect behavior, request-method restrictions, or whether request files/cookies/headers can be supplied by the LLM. Since the charter is explicit about safe mode ([charter.md](E:\projects\PentAGI\doc\charter.md:72) lines 72-74), the tool contract should be explicit too.
- The schedule already misses the deadline on paper. The design estimates `~6.7` person-days against a 6-day deadline and explicitly says “风险缓冲 = 0” ([design.md](E:\projects\PentAGI\doc\design.md:492) lines 492-506). With the bootstrap, queueing, and scope-enforcement fixes above, this is not realistically “tight but reachable.” Cut scope or accept schedule slip now.

## MEDIUM (nice-to-have)
- The CLI surface is sufficient for MVP if you count `scan`, `hint`, `list`, `report`, and `stop`; the “6 commands” story only works if `--help` is counted as a route. The docs should say that plainly, because the charter deliverables list only 4 commands while the design and plan use 5 subcommands plus help ([charter.md](E:\projects\PentAGI\doc\charter.md:35) lines 35-45; [design.md](E:\projects\PentAGI\doc\design.md:213) lines 213-244; [design.md](E:\projects\PentAGI\doc\design.md:328) lines 328-339).
- `stop` is present in the CLI contract, but the core pseudocode does not show the STOP-file check; it is only mentioned later in the implementation plan ([design.md](E:\projects\PentAGI\doc\design.md:337), [design.md](E:\projects\PentAGI\doc\design.md:550)). Keep the core algorithm and the contract aligned.
- `hints_pending.json` is a poor filename for an append-friendly queue. If a file is kept, rename it to `hints_pending.jsonl` or use per-hint spool files so the on-disk protocol is obvious.
- `exec_logs` probably also needs an index on `(task_id, created_at)` for report rendering and human timeline views, not only `(task_id, iter)` ([design.md](E:\projects\PentAGI\doc\design.md:182) lines 182-192).
- `last_heartbeat` in `tasks` is legacy Cairn terminology in a single-process CLI. `last_progress_at` would describe the field more accurately.

## SUMMARY (200-word max)
The design is directionally good but not implementation-safe yet, so I would block development start. The biggest issue is that PentAGI drops Cairn’s explicit Bootstrap phase without adding a deterministic first recon step; that makes initial fact creation, AC3, and AC6 non-deterministic. The second blocker is the hint transport: DB+file dual-write plus file flush back into DB guarantees duplicates and still leaves a read/truncate race. Third, scope control is only expressed in prompts; the tool layer does not yet enforce same-target URLs/hosts, so prompt injection or a bad LLM call can pivot scans off target. Fourth, the charter’s acceptance criteria are too weak for the stated product scope: AC4 proves report structure, not vulnerability discovery. Beyond those blockers, the schema needs a few integrity fixes (`to_fact_id` FK, stop/fail reason fields, counter checks), the snapshot dump currently lags one iteration behind the DB, and the error-code table is incomplete. Also, the schedule already exceeds the 6-day deadline with zero buffer. The MVP can work, but only after these architectural seams are tightened.
