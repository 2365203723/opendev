# Lessons · Intake

每条：**场景 → 结论 → 下次怎么办**。

---

## 1. "要漂亮" 是空话，必须给参照物

**场景**：客户说"这个页面要漂亮 / 好看 / 有设计感"。空白填这个词 = 让 Dev/Designer 猜，100% 翻车。

**结论**：IntakeAnalyst 永远要把"漂亮"展开为下列三选一，不可跳过：
- 具体参照网站 URL（"像 linear.app"）
- 客户品牌色 / 字体（"主色 #3b5bdb"）
- 流派选择（克制 / 活泼 / 商务 / 极简）

**下次怎么办**：问卷第 8 题必问；客户选不出就强推"克制单列 + 主色一色 + Inter 字体"默认值，并在 draft 第七节标为假设请老板确认。

---

## 2. 客户勾选的功能清单，实际有 30-40% 是"倍增项"

**场景**：参考 demo-blogger 项目，客户勾了"留言 + 点赞 + 阅读数 + 联系表单 + 管理后台"5 项。其中"管理后台"= 登录/权限/CSRF/会话/CRUD = 额外 3 小时。

**结论**：功能清单里**任何带"管理/后台/审核/邮件发送"字眼的项都是倍增器**，必须在 intake 阶段明确默认方案：
- 管理 → 默认"直接编辑 JSON 文件"
- 登录 → 默认"单密码 .env"
- 邮件 → 默认"只入库，不真发"
- 审核 → 默认"无审核直接显示"

**下次怎么办**：我（IntakeAnalyst）识别出倍增项 → 在 draft 第七节"假设"里列具体的简化方案 → 老板一句话可以推翻。不要直接把客户原话"做管理后台"写进功能清单。

---

## 3. 客户给了技术选型（"我想用 MongoDB"），别当约束

**场景**：客户是前同行或半懂技术的产品，会带着技术偏好来。Blog 项目的 Python 版 vs Node 版争论就是例子。

**结论**：客户的技术偏好 **≠ 硬约束**。他们的理由往往是"我上次用过"或"我老板说好"。真要上这个栈还是 Architect 评估后的决定。

**下次怎么办**：客户技术偏好只写到 draft 的第六节"备注"里，不写进第二节功能 / 第五节预算约束里。备注原话："客户提到偏好 X，请 Architect 评估是否合适"——Architect 看得见，但不被强迫。

---

## 4. "数据不能出去"≠ 全栈本地化，要拆清楚哪一层不能出

**场景**：test-client 张总说"会议内容保密，不能传到外面的服务器上"。这句话如果直接写"私有化部署"，Architect 会把所有组件都本地化（含 AI 模型），成本爆炸。

**结论**：客户说"数据不出去"时，要拆成三层问：
- 原始文件（录音/文档）不出去 → 几乎一定是硬约束
- 处理过程中的中间数据不出去 → 大概率硬约束
- 最终结果（摘要/待办）不出去 → 可能可以松动

**下次怎么办**：在问卷里把"安全/保密"拆成具体层级问，而不是一个笼统的"部署在哪"。这样 Architect 能精确判断哪些组件必须本地、哪些可以用云 API。

---

## 5. 5000 元预算 + 语音识别 + AI 总结 = 必须砍功能或砍精度

**场景**：test-client 要语音转写 + 说话人识别 + AI 摘要 + 待办提取 + 全文搜索，预算 5000。本地 Whisper 部署 + 说话人分离模型 + LLM 摘要，光调通环境就可能吃掉大量工时。

**结论**：低预算 + AI 密集型功能 = intake 阶段就要帮老板预判风险。在 draft 备注里标明"此预算下说话人识别精度可能有限"，让老板决定是砍功能还是加预算。

**下次怎么办**：遇到 AI 密集型需求 + 低预算组合时，在 gaps.md 里加一条"预算风险"标注，问卷里问老板"如果超预算，优先砍哪个功能"。
