# Lessons · Intake

每条：**场景 → 结论 → 下次怎么办**。

---

## 1. "要漂亮" 是空话，必须给参照物

**场景**：客户说"这个页面要漂亮 / 好看 / 有设计感"。空白填这个词 = 让 Dev/Designer 猜，100% 翻车。

**结论**：IntakeAnalyst 永远要把"漂亮"展开为下列三选一，不可跳过：
- 具体参照网站 URL（"像 linear.app"）
- 客户品牌色 / 字体（"主色 #3b5bdb"）
- 流派选择（克制 / 活泼 / 商务 / 极简）

**下次怎么办**：问卷第 8 题必问；客户选不出就强推"克制单列 + 主色一色 + Inter 字体"默认值，并在 draft 第七节标为假设请老板确认。

---

## 2. 客户勾选的功能清单，实际有 30-40% 是"倍增项"

**场景**：参考 demo-blogger 项目，客户勾了"留言 + 点赞 + 阅读数 + 联系表单 + 管理后台"5 项。其中"管理后台"= 登录/权限/CSRF/会话/CRUD = 额外 3 小时。

**结论**：功能清单里**任何带"管理/后台/审核/邮件发送"字眼的项都是倍增器**，必须在 intake 阶段明确默认方案：
- 管理 → 默认"直接编辑 JSON 文件"
- 登录 → 默认"单密码 .env"
- 邮件 → 默认"只入库，不真发"
- 审核 → 默认"无审核直接显示"

**下次怎么办**：我（IntakeAnalyst）识别出倍增项 → 在 draft 第七节"假设"里列具体的简化方案 → 老板一句话可以推翻。不要直接把客户原话"做管理后台"写进功能清单。

---

## 3. 客户给了技术选型（"我想用 MongoDB"），别当约束

**场景**：客户是前同行或半懂技术的产品，会带着技术偏好来。Blog 项目的 Python 版 vs Node 版争论就是例子。

**结论**：客户的技术偏好 **≠ 硬约束**。他们的理由往往是"我上次用过"或"我老板说好"。真要上这个栈还是 Architect 评估后的决定。

**下次怎么办**：客户技术偏好只写到 draft 的第六节"备注"里，不写进第二节功能 / 第五节预算约束里。备注原话："客户提到偏好 X，请 Architect 评估是否合适"——Architect 看得见，但不被强迫。

---

## 4. "数据不能出去"≠ 全栈本地化，要拆清楚哪一层不能出

**场景**：test-client 张总说"会议内容保密，不能传到外面的服务器上"。这句话如果直接写"私有化部署"，Architect 会把所有组件都本地化（含 AI 模型），成本爆炸。

**结论**：客户说"数据不出去"时，要拆成三层问：
- 原始文件（录音/文档）不出去 → 几乎一定是硬约束
- 处理过程中的中间数据不出去 → 大概率硬约束
- 最终结果（摘要/待办）不出去 → 可能可以松动

**下次怎么办**：在问卷里把"安全/保密"拆成具体层级问，而不是一个笼统的"部署在哪"。这样 Architect 能精确判断哪些组件必须本地、哪些可以用云 API。

---

## 5. 5000 元预算 + 语音识别 + AI 总结 = 必须砍功能或砍精度

**场景**：test-client 要语音转写 + 说话人识别 + AI 摘要 + 待办提取 + 全文搜索，预算 5000。本地 Whisper 部署 + 说话人分离模型 + LLM 摘要，光调通环境就可能吃掉大量工时。

**结论**：低预算 + AI 密集型功能 = intake 阶段就要帮老板预判风险。在 draft 备注里标明"此预算下说话人识别精度可能有限"，让老板决定是砍功能还是加预算。

**下次怎么办**：遇到 AI 密集型需求 + 低预算组合时，在 gaps.md 里加一条"预算风险"标注，问卷里问老板"如果超预算，优先砍哪个功能"。

---

## 6. 客户有现成 demo，"修 bug"和"新建服务"容易混淆

**场景**：xinyue 客户有一个本地跑通的 RVC 变声 demo，要求"改成 Web 版"。客户描述里混着"修 demo 的 bug"和"做多用户 Web 服务"两件事，如果不拆开，Dev 会不知道哪些是修复、哪些是新建。

**结论**：客户有 demo 时，intake 阶段必须把需求拆成两类：
- 修复类（demo 已有但有问题）→ 列具体 bug 现象
- 新建类（demo 没有的能力）→ 列功能清单

**下次怎么办**：在 facts.md 里专门开一节"demo 现状 vs 目标差距"，把修复项和新建项分开列。Dev 看到的功能清单里，修复项标 [FIX]，新建项标 [NEW]，不混在一起。

---

## 7. 预算/截止未定时，功能范围仍可先锁定

**场景**：xinyue 客户在问卷里没有填预算和截止日期，但功能需求描述得很清楚。如果等预算确认才出草稿，会浪费一个来回。

**结论**：功能范围和预算/截止是两个独立维度，可以解耦处理。功能范围先锁定，预算/截止在草稿里标"待定，开工前必须确认"，不阻塞功能确认流程。

**下次怎么办**：draft 第四节"预算与截止"单独列，标注"待定"并加警告"开工前必须确认，否则无法排期报价"。功能清单照常写完，不等预算。

---

## 9. 安全工具类项目：合规免责声明是 intake 阶段的硬性输出

**场景**：PentAGI 客户要做 Agent 自动化渗透测试工具，功能需求清晰，但原始材料里完全没提合规/免责。如果直接写进功能清单，Dev 可能忽略这个非功能需求。

**结论**：凡是涉及渗透测试、漏洞扫描、安全工具的项目，intake 阶段必须在 draft 里明确写入：
- 验收标准里加一条"README 包含免责声明（仅用于授权测试环境）"
- 第六节备注里提醒 Dev 代码中加免责声明
- 不做的事里加"不做未授权目标的扫描防护"（这是用户责任，不是工具责任）

**下次怎么办**：识别到安全工具类项目时，在 gaps.md 里加"合规风险"标注，并在 draft 验收标准里强制加一条免责声明相关的布尔检查项。

---

## 10. 客户指定开源项目参考，intake 阶段要抓架构洞察而非功能列表

**场景**：PentAGI 客户给了 Cairn（github.com/oritera/Cairn）作为参考，只说"用到我的项目里"。如果只记录"参考 Cairn"，Architect 拿到需求时仍然不知道复用哪些、丢弃哪些。

**结论**：客户给开源项目参考时，intake 阶段必须抓取 README + 核心源文件，提炼出：
- 核心架构模式（黑板架构、OODA 循环、三原语）
- 哪些组件适合 MVP 复用（数据模型、schema）
- 哪些组件太重需要简化（Docker Dispatcher、分布式 Worker）
- 与客户需求的契合点和差异点

这些洞察写进 facts.md，让 Architect 直接拿来用，不用重新研究。

**下次怎么办**：问卷里加一题"你给的参考项目，哪些功能是你最想要的？"；同时在 facts.md 里专门开一节"参考项目架构洞察"，抓取 README 后填入。

---

## 8. GPU 密集型实时服务有物理延迟下限，intake 阶段就要告知

**场景**：xinyue 要求实时变声延迟 ≤ 500ms。RVC 推理 + 网络 RTT 的物理下限约 150ms，但客户可能期望"像电话一样"的 < 50ms。如果 intake 阶段不说清楚，Dev 交付后客户会觉得"不够实时"。

**结论**：凡是涉及 GPU 推理 + 实时流的需求，intake 阶段就要在草稿里写"技术风险告知"，明确延迟下限和并发瓶颈，让客户在签字前知情。

**下次怎么办**：draft 里加第六节"技术风险告知"，列出物理约束（延迟下限、并发上限、硬件依赖）。这些不是 Dev 的锅，是物理限制，客户需要提前接受。
