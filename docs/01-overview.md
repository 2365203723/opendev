# 01 · 系统概览

> 你是第一次接触这套系统？这份文档告诉你它是什么、为什么这么设计、核心能力边界。读完需要 10 分钟。

---

## 一、我们在解决什么问题

**场景**：一个接项目的开发者（或小团队），想把 AI coding agent 从"单次对话"升级成"一家持续运转的公司"。每天有接单、拆任务、开发、质检、交付的完整流程，交付完能沉淀经验给下次用。

**传统做法的痛点**：
- Claude Code 单会话上下文有限，长任务频繁漂移
- 每次接新项目都要重新解释背景，知识不沉淀
- 单一 agent 身兼多职，质量参差
- 没有可追踪的工单流程，难复盘

**本方案给的答案**：
- **虚拟公司编制**：五个角色各司其职（CEO / PM / Architect / Dev / Reviewer），互相交接
- **文件即记忆**：所有 agent 通过文件 + 工单通信，不依赖上下文传递
- **Paperclip 工单平台**：给每个任务分配身份、状态、责任人，像真公司的 OA
- **跨项目经验库**：每次项目结束写入 lessons/，下次自动受益
- **资产集中化**：所有可复用资产在 H 盘，Git 版本化，随时迁移

## 二、五层架构图

```
┌─────────────────────────────────────────────────────────────┐
│ L5  业务层                                                   │
│     5 角色流水线：CEO → PM → Architect → Dev → Reviewer     │
│     Soul/Heartbeat 定义行为，工单流推进状态                 │
├─────────────────────────────────────────────────────────────┤
│ L4  编排层                                                   │
│     Paperclip 本地工单平台（HTTP :3100 + Postgres :54329）   │
│     Company → Project → Goal → Issue → Comment 五级实体      │
├─────────────────────────────────────────────────────────────┤
│ L3  Agent 定义层                                             │
│     每个 agent 四件套：soul.md / heartbeat.md                │
│                       / tools.md / agents.md                 │
├─────────────────────────────────────────────────────────────┤
│ L2  资产仓库层                                               │
│     H:\claude-assets\                                        │
│       ├── agents/ skills/ plugins/                           │
│       ├── companies/web-outsource/…（公司编制）              │
│       ├── skeletons/ snippets/ lessons/（可复用经验）        │
│       └── mcp-servers/（自建 MCP）                           │
├─────────────────────────────────────────────────────────────┤
│ L1  配置/接入层                                              │
│     ~/.claude/{settings.json, agents→软链, memory/MEMORY.md} │
│     MCP 接入：Multi-CLI / chrome-devtools / playwright /     │
│              context7 / fetch / memory / paperclip-mcp        │
├─────────────────────────────────────────────────────────────┤
│ L0  物理层                                                   │
│     7 个分区各司其职（C 总部 / E 项目 / G 报告 / H 资产…）   │
└─────────────────────────────────────────────────────────────┘
```

每一层只依赖下一层，上层换掉不影响下层。比如：

- 把 Paperclip 换成 Linear / Jira（改 L4），不用动物理分区和 agent 定义
- 把公司从 `web-outsource` 换成 `data-team`（改 L5），不用动 Paperclip
- 加一块新硬盘（改 L0），只需更新 L2 软链指向即可

## 三、物理分区规划

7 个分区按"办公楼"类比分配：

| 盘 | 角色 | 用途 | 使用特征 |
|---|---|---|---|
| **C** | 总部 | `~/.claude/` 运行时配置、memory | Claude 写死路径，不能动 |
| **D** | 下载区 | 原始素材、下载缓存 | 网络写入 |
| **E** | 研发中心（热） | `inbox/`、`projects/` 活跃项目 | 频繁读写，SSD 最优 |
| **F** | 设计素材库 | 备用（图标字体模板） | 大量小文件 |
| **G** | 质量部 | `qa-reports/` 测试报告 | 只增不改 |
| **H** | 知识中心 ⭐ | `claude-assets/` + `paperclip/` + `logs/` | 大量结构化数据，最重要 |
| **I** | 档案室 | `archive/` 交付后的冷数据 | 读多写少 |
| **J** | 沙盒 | 风险操作、临时容器 | 随时可清 |

**为什么是 7 个**：
- 分区隔离让不同类型的数据互不影响（写报告不会污染项目目录）
- 备份策略可分层（H 盘周备份，E 盘日备份，G 盘月备份）
- 某一块坏了不会全盘丢失

**最少配置**：如果只有 C 盘 + 一块额外盘，也能跑。把 E/F/G/H/I/J 都合并到同一块大盘，只是失去分区隔离优势。

## 四、核心设计理念（7 条）

理解这 7 条，就理解了整个系统的设计哲学。

### 1. 文件即记忆，上下文不传递

所有 agent 只通过"写文件 + 工单 Comment 引用路径"通信。CEO 把章程写到 `doc/charter.md`，PM 只看到一句话"读 charter.md"，自己去读。

**好处**：每个 agent 的上下文干净，不会因为上游 agent 的长篇叙述而挤占自己的思考空间。

### 2. 四件套分离身份与行为

每个 agent 有 4 个文件：
- **soul.md**（我是谁）
- **heartbeat.md**（被唤醒时我先做什么）
- **tools.md**（我被允许做什么）
- **agents.md**（我可以呼叫谁）

价值观极少改（季度级），动作路径常改（周级），分开文件让修改影响面清晰。

### 3. Paperclip 双角色（调度器 + 记账簿）

| 模式 | 谁在执行 | 适用场景 |
|---|---|---|
| **记账簿模式** | 用户在 Claude 会话里扮演 agent | MVP 阶段主流，调试成本低 |
| **调度器模式** | Paperclip 心跳触发 agent 自主工作 | 成熟后无人值守，夜间跑 |

两种模式共用同一套 agent 定义、同一个工单平台，切换成本低。

### 4. 三层配置覆盖（类 Git）

```
全局层：~/.claude/          共用资产（agent / skill / MCP）
项目层：<项目>/.claude/     项目专属（覆盖同名）
本地层：settings.local.json 个人专属（不入库）
```

下层覆盖上层。通用的 `code-reviewer` 放全局，某项目的"React 代码 reviewer"放项目层。

### 5. 资产集中 + 软链接

所有可复用资产在 `H:\claude-assets\`，C 盘用 `mklink /D` 软链过去。

**好处**：
- 资产可独立 git 版本化
- 换机器时只需备份 H 盘
- C 盘不被大量小文件污染

### 6. Lessons 库 = 跨项目 checkpoint

每次项目结束，每个角色写一份"踩坑笔记"到 `lessons/{角色}.md`。下次同角色项目第一步先读自己的 lessons。

**关键**：经验复用不靠"记住"，靠"读"。这让这套系统**可持续积累**。

### 7. 多模协审（Multi-CLI）

Dev 完工后可调用 `Ask-Codex` + `Ask-Gemini` + `Ask-OpenCode` 让其他模型审视代码，覆盖单模型盲点。Reviewer 的审查权威性比常规 code review 更稳。

## 五、5 角色流水线

```
客户需求（E:\inbox\{客户}\requirements.md）
    │
    ▼
┌─────────────────────────────────────────────┐
│ CEO-Orchestrator（总经理）                   │
│   读需求 → 写章程（布尔验收 10 条）          │
│   创建 Paperclip Project + Goal              │
│   下 kickoff 工单给 PM                       │
└─────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────┐
│ PM-Planner（项目经理）                       │
│   读章程 + lessons/pm.md                     │
│   拆 N 个工单（附 mermaid 依赖图）           │
│   分配给 architect / dev / reviewer          │
└─────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────┐
│ Architect（架构师）                          │
│   技术选型 + 数据模型 + API 契约             │
│   落盘 doc/design.md                         │
│   有二选一决策权（MVP 阶段授权）             │
└─────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────┐
│ Frontend/Backend-Dev（开发）                 │
│   实施 → Multi-CLI 双模协审 → 10 条自检      │
│   落盘 src/ + README.md                      │
└─────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────┐
│ Reviewer（质检）                             │
│   代码 + 安全 + 视觉 + 性能 四维审查         │
│   落盘 G:\qa-reports\{项目}\review-NNN.md    │
│   结论：PASS / FAIL / PASS_WITH_WARNINGS     │
└─────────────────────────────────────────────┘
    │
    ▼
CEO 最终验收 → 归档 I:\archive\{年}\{项目}
FAIL 回环 → reassign 给 dev（最多 5 轮）
```

**每个角色的"下班条件"统一**：产出指定文件 + 关闭工单 + 留交接 comment。

## 六、这套系统能做什么，不能做什么

### ✅ 能做

- 一次性接一个中小型交付项目（1 天到 1 周工期）
- 同时跑多个项目（每个项目独立 `E:\projects\{项目}` 目录）
- 沉淀跨项目经验（lessons 库）
- 回溯任何历史项目的完整交付链（Paperclip 工单 + Git history）
- 多虚拟公司并存（web-outsource / data-team / mobile-team）

### ❌ 暂不能做

- **真正无人值守的心跳调度**（Paperclip 调度器当前未自动触发 agent run，需手动扮演）
- **多用户协作**（单机单用户架构）
- **实时监控 dashboard**（Paperclip Board 是静态 UI，不是实时流）
- **自动回归测试**（目前只靠 reviewer 人为判断）

### 🚧 待优化

- Chrome DevTools MCP 白名单只到 C 盘（截图要中转）
- Paperclip goal/project 的 PATCH 契约需摸索
- 自动化 lessons 抽取 hook 未实现
